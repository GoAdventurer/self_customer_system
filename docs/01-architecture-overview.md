# 企业级智能客服系统 · 架构设计方案

> 面向大型电商企业的全渠道、多租户、人机协同智能客服系统。本文档为**架构方案设计**，不含具体实现代码。

---

## 1. 设计背景与目标

大型电商场景的客服诉求具有以下特征：高并发（大促峰值可达日常 10~50 倍）、长尾业务多（退换货、发票、价保、物流、营销活动规则）、强合规要求（隐私、资金、消保）、以及对"一次性解决率"和"成本"的双重压力。

### 1.1 业务目标
- **降本**：机器人独立解决率（自助率）≥ 70%，复杂问题人机协同处理。
- **提效**：平均首响 < 3s，端到端会话解决时长下降 40%。
- **提质**：意图识别准确率 ≥ 92%，工单/退款等关键流程零误操作。
- **可控**：成本、隐私、合规可观测、可审计、可回溯。

### 1.2 设计原则
- **分层解耦**：每层职责单一，通过契约（接口/事件）通信，可独立演进与扩缩容。
- **编排优先**：所有跨系统副作用（退款、改单、写 CRM）走显式编排 + 幂等 + 补偿，而非让模型"自由调用"。
- **混合智能**：规则、分类器、检索、大模型分工协作，按"够用即可"的最低成本模型完成任务。
- **人机协同**：模型负责理解与起草，高风险动作由人确认或由确定性流程兜底。
- **可观测可治理**：全链路埋点、可灰度、可回滚、可审计。

## 3. 分层详细设计

### 3.1 接入网关层（新增横切层）
- **能力**：协议适配（HTTP/WebSocket/gRPC/电话 SIP）、鉴权与会话票据、租户路由、限流熔断、灰度分流。
- **BFF**：为不同渠道（App / Web / 小程序 / 客服工作台）提供差异化聚合接口。
- **大促保护**：分级限流（按租户/渠道/接口）、排队与降级（高峰期优先放行 VIP 与资金类会话）。

### 3.2 ① 输入层
- **职责**：接收并标准化文本、语音、表单、图片/视频、文档附件等全格式输入。
- **细化设计**：
  - 注入会话上下文：`session_id`、`channel`、`tenant_id`、`user_profile`（等级、历史订单、风险标签）。
  - **语音**：流式 ASR（VAD 断句 + 标点恢复 + 行业热词纠错），输出带时间戳的文本流。
  - **多模态**：图片走 OCR / 视觉理解（如商品破损图、面单截图），结构化后并入上下文。
  - **预处理**：脱敏（手机号/身份证/卡号掩码）、敏感词、语种识别、归一化（全半角、单位）。
  - **知识热更新触发**：识别"活动规则变更/新政策"类事件，触发知识库增量索引。

#### 3.2.1 多格式文档解析能力

用户可能通过聊天窗口发送各类文档附件（发票截图、退货凭证、合同PDF、邮件转发等），系统需统一解析为结构化上下文。

**支持格式与解析策略**：

| 格式 | 解析引擎 | 输出结构 | 典型场景 |
|------|---------|---------|---------|
| PDF | PyMuPDF + 表格识别(Camelot/Tabula) | 文本段落 + 表格JSON + 元数据 | 合同、发票、政策文件 |
| Word (.docx) | python-docx 提取 + 结构保留 | 段落/标题层级 + 表格 + 嵌入图片 | 投诉函、退换货申请表 |
| HTML/邮件 | 正文提取(Readability算法) + 附件递归 | 纯文本 + 链接 + 附件列表 | 邮件转发、网页截图 |
| Excel (.xlsx) | openpyxl 结构化读取 | 表格JSON(含公式计算值) | 批量订单、对账单 |
| 图片 (JPG/PNG) | OCR(PaddleOCR) + 版面分析 + 视觉理解 | 文字区域 + 表格区域 + 图像描述 | 面单、破损图、聊天截图 |
| 语音附件 (mp3/wav) | ASR转写 + 说话人分离 | 带时间戳文本 + 角色标注 | 录音投诉、电话回放 |

**解析流水线设计**：

```
用户上传附件
    │
    ▼
┌─────────────────────────────────────────────────────────────────┐
│ 文档解析网关 (Document Parser Gateway)                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Step 1: 格式检测与安全扫描                                      │
│   · MIME类型识别(不信后缀,检测magic bytes)                       │
│   · 文件大小限制: 单文件≤20MB, 单会话累计≤50MB                   │
│   · 病毒/恶意文件扫描(ClamAV)                                   │
│   · 嵌入脚本/宏检测(Office宏→拒绝解析,仅提取纯内容)             │
│                                                                 │
│  Step 2: 路由到对应解析器                                        │
│   · PDF → PDF解析器(文本层优先,无文本层→OCR降级)                 │
│   · 图片 → OCR管线 + 可选视觉大模型理解                         │
│   · Office → 结构化提取器                                       │
│   · 其他 → 通用文本提取(Apache Tika兜底)                        │
│                                                                 │
│  Step 3: 结构化输出 + PII脱敏                                    │
│   · 提取的文本走统一脱敏管线(与文字输入相同)                     │
│   · 表格保留结构(JSON schema)                                   │
│   · 关键字段自动标注(金额/日期/单号/姓名)                       │
│   · 输出统一格式: ParsedDocument { sections, tables, metadata }  │
│                                                                 │
│  Step 4: 上下文注入                                              │
│   · 解析结果拼接到会话上下文(带来源标记)                         │
│   · 超长文档→摘要压缩(保留关键信息,控制token)                   │
│   · 表格数据→结构化存储(可供执行层精确查询)                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**性能与降级**：
- 解析超时：单文档解析上限 30s，超时返回"文档处理中"异步通知结果
- 解析失败：降级为文件名+大小信息，建议用户用文字描述内容
- 大促降级(L2+)：关闭文档解析通道，仅接收文本输入
- 异步解析：大文档(>5MB)走异步队列，先响应"已收到，正在处理"

### 3.3 ② 意图层
- **职责**：混合识别意图 + 二级槽位抽取。
- **混合识别（三级)**：
  1. **规则/正则**：高确定性场景（"查物流""开发票"）直接命中，零成本零延迟。
  2. **分类器**：轻量文本分类（BERT 类/小模型）覆盖常见意图，输出置信度。
  3. **函数调用（LLM）**：长尾、复合、模糊表达由大模型做意图判定与工具选择，兜底。
- **置信度路由**：高置信走规则/分类器；低置信或多意图升级到 LLM；不确定时澄清式反问。
- **槽位抽取**：订单号、手机号、发票抬头、税号、投诉原因、商品名称、投诉类型、期望处理方式等结构化字段；带校验（订单号格式、订单归属校验）与缺失槽位追问策略。
- **意图体系**：FAQ / 工单 / 投诉 / 退换货 / 价保 / 物流 / 发票 / 营销咨询 / **导购推荐** / 转人工，支持层级化标签与多租户自定义。

#### 3.3.1 话题管理与话题栈（复杂话题表达）

单意图识别之上引入**话题（Topic）**层概念，解决"一句话含多话题"与"会话中来回切换话题"的复杂表达问题（如：先问货品功能 → 转去查订单 → 又回来问刚才那款货）。

**意图与话题的关系**：意图是单轮的动作判定，话题是跨轮的语义焦点。一个话题可包含多轮多意图（如"导购"话题内含介绍、比价、加购）。

**话题栈（Topic Stack）结构定义**：

```jsonc
TopicStack {
  active_topic: TopicFrame,        // 栈顶, 当前焦点话题
  stack: TopicFrame[],             // 被挂起的话题(后进先出)
  history: TopicRef[],             // 已关闭话题(用于"刚才那个"指代消解)
  max_depth: 5                     // 防止无限嵌套, 超深则归并最旧话题
}

TopicFrame {
  topic_id: string,
  topic_type: "PRODUCT" | "ORDER" | "AFTER_SALE" | "LOGISTICS" | "COMPLAINT" | "PROMOTION" | ...,
  focus_entity: object,            // 话题焦点实体, 如 {product_id, order_id}
  slots: map,                      // 该话题已收集的槽位(话题隔离, 不串话)
  status: "ACTIVE" | "SUSPENDED" | "RESOLVED",
  related_graph_run: RunId | null, // 若已进入规划层流程, 关联其 GraphRun
  last_active_at: timestamp
}
```

**话题切换决策**：

| 用户行为 | 检测信号 | 栈操作 |
|---------|---------|--------|
| 开启新话题 | 新意图 + 新焦点实体，与栈顶不相关 | `push` 新 TopicFrame，旧话题置 SUSPENDED |
| 延续当前话题 | 意图/实体与栈顶一致或为其子动作 | 更新栈顶 slots/focus |
| 回切旧话题 | 指代"刚才/上一个" 或显式提及旧实体 | 从 stack/history `恢复`，重新置顶 |
| 话题完成 | 流程结束 / 用户确认满意 | `pop` 栈顶 → 自动恢复下一个 SUSPENDED 话题 |
| 多话题同句 | `intent_cnt > 1`，LLM 拆解为多话题 | 依次入栈，逐个处理并提示用户 |

**指代消解**：结合 `history` + `focus_entity`，把"它/这款/刚才那个订单"解析回具体实体，再补入当前槽位；解析失败则澄清式反问。

**与各层协同**：话题栈持久化在 `GraphState`（新增 `topic_stack` 通道，reduce=`last`）；话题挂起=对应 GraphRun 进入 `WAITING`；话题恢复=按 RunId 续跑（见 `orchestration-graph-design.md` §8 断点恢复）。

### 3.4 ③ 规划层
- **职责**：用 LangGraph 将意图映射为**任务依赖图（DAG）**，做确定性流程编排。
- **示例（退款）**：`检索凭证 → 校验订单状态 → 风控校验 → 发起退款单 → 回写 CRM → 通知用户`。
- **关键设计**：
  - **状态机 + DAG 混合**：长流程用状态机持久化（可中断、可恢复、可人工介入）。
  - **条件分支与回环**：根据校验结果分流（如风控不通过转人工）。
  - **人确认节点（HITL）**：资金/不可逆动作插入人工审批或用户二次确认节点。
  - **流程版本化**：每个业务流程图可灰度、可回滚，运营可在低代码编排台维护。

> **编排图结构（点边）详细设计**见 `orchestration-graph-design.md`：包含图定义/全局状态（通道+归并器）/节点/边四类核心结构定义、12 类节点与 10 类边的设计、条件谓词、退款流程完整点边范例、Saga 补偿语义、图校验规则与断点恢复。

### 3.5 ④ 执行层
- **职责**：调用外部能力并保障跨系统稳定落地。
- **能力模块**：RAG 检索、CRM 读写、工单 API、退款/支付网关、表单填写、票据/发票生成、物流查询、营销规则引擎。
- **可靠性设计（核心）**：
  - **幂等**：所有写操作携带幂等键（`session_id + step_id + payload_hash`）。
  - **重试与退避**：可重试错误指数退避；不可重试错误快速失败并补偿。
  - **Saga 补偿**：跨系统事务失败时按逆序补偿（如退款已发起但 CRM 回写失败的对账修复）。
  - **超时与熔断**：每个下游设独立超时与熔断阈值，故障隔离。
  - **统一工具网关（Tool Gateway）**：对外部系统做协议适配、鉴权、审计、Mock，便于测试与替换。

**CRM 集成核心设计**：

| 维度 | 设计方案 | 说明 |
|------|---------|------|
| **鉴权** | OAuth2.0 Client Credentials + Token缓存(提前5min刷新) | 避免每次请求重新鉴权 |
| **字段映射** | 配置化映射引擎(YAML DSL) + 运行时热加载 | 不同CRM字段名/类型不同，映射可配 |
| **数据同步** | 双写模式(实时) + 对账任务(T+1校验) | 实时保体验，对账保一致性 |
| **版本兼容** | API版本锁定 + 兼容层(Adapter Pattern) | CRM升级不影响客服系统 |
| **灰度切换** | 按租户/流量比例灰度从旧CRM迁移到新CRM | 验证通过再全量切换 |
| **故障隔离** | CRM不可用时→工单暂存本地队列→恢复后补写 | 客服流程不因CRM故障阻塞 |
| **Mock模式** | 开发/测试环境自动路由到Mock Server | 不依赖真实CRM即可联调 |

**字段映射引擎示例**：

```yaml
# crm-field-mapping.yaml (可热更新,无需重启)
salesforce:
  ticket_create:
    mapping:
      Subject: "$.intent_label + ' - ' + $.user_name"
      Description: "$.conversation_summary"
      Priority: 
        transform: "map"
        rules:
          HIGH: "Urgent"
          MEDIUM: "High" 
          LOW: "Medium"
      Contact__c: "$.user_id"
      Order_Number__c: "$.slots.order_id"
      Channel__c: "$.channel"
    required_fields: [Subject, Description, Priority]
    default_values:
      Status: "New"
      Origin: "AI_Customer_Service"
      
custom_crm:
  ticket_create:
    endpoint: "/api/v2/tickets"
    method: POST
    mapping:
      title: "$.intent_label"
      content: "$.conversation_summary"
      priority: "$.risk_level"
      customer_id: "$.user_id"
      metadata:
        session_id: "$.session_id"
        ai_confidence: "$.intent_confidence"
```

**CRM API 调用链路**：

```
规划层(DAG节点) → 执行层(Tool Gateway)
    │
    ├── 1. 字段映射: 内部模型 → CRM字段 (映射引擎)
    ├── 2. 鉴权注入: 获取/刷新OAuth Token
    ├── 3. 请求发送: HTTP POST + 幂等键Header
    ├── 4. 响应解析: CRM响应 → 内部模型 (反向映射)
    ├── 5. 审计记录: 请求/响应/耗时/结果 写入审计日志
    │
    ├── [成功] → 返回CRM工单ID → 后续节点继续
    ├── [限速429] → 等待Retry-After → 重试
    ├── [超时] → 指数退避重试(最多3次) → 仍失败→本地暂存队列
    └── [业务错误] → 解析错误码 → 快速失败 → 补偿/人工
```

**多CRM并存场景(多租户)**：

```

租户B → 自研CRM (国内业务)

路由规则: tenant_config.crm_provider → 选择对应适配器
迁移过程: 租户B 从自研CRM → Salesforce 灰度迁移
  · Phase1: 10%流量双写(旧写+新写), 对账验证
  · Phase2: 50%流量切新, 旧只读
  · Phase3: 100%切新, 旧下线
```

### 3.6 ⑤ 推理层
- **职责**：根据复杂度与成本/隐私约束做模型路由。
- **路由策略**：
  - **复杂度路由**：简单问答/改写 → 小模型；复杂申诉/多轮推理 → 大模型。
  - **本地 vs 云端**：含敏感数据（身份、资金、隐私）走本地/私有化模型；通用内容可走云端，控成本保隐私。
  - **能力路由**：函数调用强的模型做工具编排，长文本模型做长会话/长文档摘要。
- **成本与性能优化**：
  - **语义缓存**：高频 FAQ 答案缓存（精确 + 语义相似命中）。
  - **Prompt/上下文压缩**：会话摘要 + 检索片段裁剪，控制 token。
  - **降级**：大模型不可用/超载时回退到小模型或预设话术，保证可用性。
  - **流式输出**：首 token 优先，提升体感首响。

#### 3.6.1 多模型路由选型矩阵

系统接入多家模型厂商，按能力、成本、延迟、隐私四个维度做精细化路由，实现成本最优。

- **职责**：采集质量信号，驱动迭代闭环。

  - **离线评测集**：维护回归评测集，每次模型/Prompt/知识变更跑回归，防退化。

## 4. 横切能力设计

### 4.1 会话状态管理
- 短期会话上下文（Redis，TTL）+ 长流程状态持久化（DB，可恢复）。
- 多轮对话状态、当前所处流程节点、已填槽位、待确认事项统一管理。

### 4.2 记忆与知识库（RAG）
- **知识分层**：FAQ 库、政策/规则库、商品库、订单/工单实时数据、历史会话经验库。
- **检索架构**：向量检索 + 关键词（BM25）混合 + Rerank；结构化数据走精确查询而非向量。
- **知识治理**：版本管理、生效时间、来源可溯、冲突检测；运营可在知识后台增删改并触发增量索引（呼应输入层的热更新）。
- **防幻觉**：答案强制附引用来源；无据可查时不臆答，转澄清或人工。
- **记忆分层**：会话内记忆（短期）/ 用户画像记忆（长期）/ 组织级经验（沉淀复用）。

#### 4.2.1 商品导购与推荐能力

"货品功能介绍"是检索型问答，但电商场景常延伸到**导购、推荐、比价、搭配、引导下单**，这超出 FAQ/RAG 的范畴，需独立能力模块。把商品域细分为三类能力：

| 能力 | 性质 | 实现路径 | 触发意图 |
|------|------|---------|---------|
| 商品问答 | 检索生成 | 商品库 RAG（规格/参数/卖点）+ 生成附引用 | FAQ / 导购推荐 |
| 导购推荐 | 推荐策略 | 召回（向量+标签+协同过滤）→ 排序（CTR/转化/利润）→ 生成推荐理由 | 导购推荐 |
| 比价与搭配 | 多商品对比/组合 | 多商品规格结构化对比 + 搭配规则引擎 + LLM 整合表达 | 导购推荐 |

**推荐链路（执行层新增工具节点）**：

```
用户需求(画像+话题focus_entity+约束如预算/人群)
   → 召回 (向量相似 + 标签过滤 + 协同过滤, Top-N)
   → 过滤 (库存/上下架/合规/不可推商品)
   → 排序 (转化率 × 利润 × 个性化, 可配权重)
   → 生成 (LLM 产出推荐理由, 强制基于真实商品属性, 防夸大)
   → 输出 (商品卡片 + 理由 + 比价表, 附"加购/详情"入口)
```

**导购防风险约束**：推荐理由只能基于商品库真实属性（防虚假宣传/夸大功效）；价格/促销必须实时取数（防过期信息）；医疗/食品等品类加合规话术与免责。

#### 4.2.2 客户情感画像（长期情感记忆）

情绪信号不止用于单会话升级（见深化推演场景2），还沉淀为**长期情感画像**，影响后续服务策略：

```jsonc
EmotionProfile {           // 写入长期记忆(用户画像层)
  user_id: string,
  baseline_sentiment: float,        // 历史基线情绪(识别"易怒型"客户)
  recent_trend: "IMPROVING" | "STABLE" | "DETERIORATING",
  sensitive_topics: string[],       // 历史触发负面情绪的话题(物流/退款...)
  escalation_history: int,          // 历史升级次数
  preferred_tone: "EMPATHETIC" | "EFFICIENT" | "FORMAL",  // 偏好的服务语气
  churn_risk: "LOW" | "MEDIUM" | "HIGH"  // 流失风险(负面情绪+低活跃)
}
```

**画像驱动的策略联动**：
- **模型路由**：高敏感/易怒客户 → 优先大模型（共情能力强）、降低升级阈值。
- **话术风格**：按 `preferred_tone` 调整生成风格（高效型少寒暄，共情型多安抚）。
- **主动服务**：`churn_risk=HIGH` 触发挽留流程（优惠/专属客服）。
- **闭环**：情感画像随每次会话更新，进入反馈层数据飞轮（见 §3.7）。

### 4.3 人机协同（Human-in-the-Loop）
- **转人工策略**：用户主动请求、低置信、高风险动作、负面情绪升级、连续未解决。
- **客服工作台**：模型起草回复 + 关键信息卡片（订单/风险/建议动作），人工一键确认或修改。
- **协同模式**：Copilot（辅助人工）与 Autopilot（机器人主导）按场景与风险分级切换。

### 4.4 安全与合规
- **数据安全**：传输 TLS、存储加密、字段级脱敏、最小权限。
- **隐私合规**：PII 识别与掩码、数据分级、留存与删除策略（符合个保法/GDPR）。
- **资金安全**：退款/改价等动作强制风控校验 + 额度阈值 + 审批。
- **审计**：全量动作可审计追溯（谁/何时/对哪个会话/做了什么）。
- **提示注入防护**：对用户输入与检索内容做注入检测，工具调用做权限校验与参数校验。

### 4.5 可观测性
- **Tracing**：全链路 trace（会话→意图→规划→执行→模型调用），定位慢节点与失败点。
- **Metrics**：业务（解决率/转人工率）+ 技术（QPS/延迟/错误率/token 成本）。
- **Logging**：结构化日志 + 会话回放，支持坏案例复盘。
- **告警**：模型可用性、下游故障、成本异常、质量指标跌破阈值。

## 6. 非功能性需求（NFR）

- **可用性**：核心链路 ≥ 99.9%，模型与下游均有降级方案，单点故障不致全局不可用。
- **扩展性**：各层无状态服务水平扩展；状态外置；大促可弹性扩容。
- **性能**：首响 < 3s（流式），简单问答 P95 < 1.5s。
- **一致性**：跨系统副作用最终一致（Saga + 对账），关键资金操作强一致校验。
- **成本**：模型路由 + 缓存 + 压缩，单会话成本可观测、可设预算上限。

## 7. 部署与技术栈（参考）

- **编排**：LangGraph（任务图/状态机）。
- **服务**：微服务 + API 网关，容器化（K8s）部署，HPA 弹性伸缩。
- **模型**：本地私有化模型（敏感/高频）+ 云端大模型（复杂/长尾），统一模型网关接入。
- **中间件**：消息队列（异步执行、削峰）、缓存、向量库、搜索引擎。
- **环境**：开发/灰度/生产隔离，流程与 Prompt 支持灰度发布与一键回滚。

## 8. 演进路线（建议）

1. **阶段一（MVP）**：单租户、文本渠道、FAQ + 工单 + 转人工，规则 + 小模型 + 基础 RAG。
2. **阶段二（增强）**：多渠道（语音/多模态）、复杂流程编排（退款/价保）、模型路由与缓存、人机协同工作台。
3. **阶段三（规模化）**：多租户、数据飞轮闭环、全面可观测与质检、成本与合规治理体系化。
