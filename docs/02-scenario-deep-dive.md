# 企业级智能客服系统 · 架构深化推演

> 基于主架构方案（readme.md），通过四个核心业务场景的完整时序推演，验证架构可行性并补充落地细节。

---

## 1. 推演方法论

### 1.1 场景选取原则

选取四个场景，各自验证架构的不同能力维度，互补覆盖全部核心机制：

| 场景 | 核心验证点 | 复杂度 |
|------|-----------|--------|
| 退款全流程 | Saga/幂等/补偿/HITL/状态机 | 跨层最深、副作用最多 |
| 投诉升级 | 情绪检测/升级策略/人机协同/质检 | 多轮对话、非确定性决策 |
| 大促流量峰值 | 限流/降级/弹性/熔断/缓存 | NFR极限压力 |
| FAQ快速应答 | 模型路由/语义缓存/RAG/防幻觉 | 成本与质量平衡 |

### 1.2 推演深度标准

每个场景包含：
- **完整时序图**：从用户请求到最终响应，标注每层模块交互
- **关键设计决策**：带推理依据的技术选型
- **异常分支**：穷举故障点 + 处理策略 + 补偿动作
- **量化指标**：延迟、成本、覆盖率等可度量数据

## 2. 场景1：退款全流程

### 2.1 场景描述

用户通过 App 发起"我要退这个订单的钱"，订单金额 ¥299，已签收 3 天，在退货退款时间窗内。此场景跨层流转最深（6 层全涉及），副作用最多（支付/CRM/工单/通知），最能检验 Saga 补偿与幂等设计。

### 2.3 关键设计决策

| # | 决策点 | 设计选择 | 推理 |
|---|--------|---------|------|
| D1 | 意图识别走哪一级？ | "退钱"→ 规则正则直接命中（conf≥0.9），跳过分类器和LLM | 高确定性场景零成本，P95 < 5ms |
| D2 | 槽位缺失怎么办？ | 追问策略：先尝试从 user_profile 自动补全（最近订单），失败再追问 | 减少交互轮次，降低用户流失 |
| D3 | DAG 中 T4(用户确认) 在哪？ | 在风控通过后、实际退款前 | 不可逆操作前必须有人确认，满足合规 |
| D4 | T5 退款的幂等键 | `session_id + "REFUND" + order_id + amount` | 防止用户重复点击或网络重试导致重复退款 |
| D5 | T6/T7 并行还是串行？ | 并行执行，互不依赖 | CRM 回写失败不应阻塞用户通知 |
| D6 | 回复生成用大模型还是模板？ | 模板 + 变量填充（金额、到账时间） | 确定性场景无需 LLM 生成，降成本 |

### 2.4 异常分支与补偿

| 异常点 | 检测方式 | 处理策略 | 补偿动作 |
|--------|---------|---------|---------|
| T1 订单不存在 | OrderService 404 | 反馈"订单号有误"，重新追问 | 无（无副作用） |
| T2 超出退款时限 | RuleEngine 拒绝 | 反馈"已超出退款期限"，建议转人工申诉 | 无（无副作用） |
| T3 风控拦截 | RiskService 拒绝 | 标记高风险，自动转人工审核组 | 无（无副作用） |
| T5 支付网关超时 | 30s 无响应 | 重试(指数退避,最多3次)；仍失败则挂起+人工 | 无（幂等键保安全） |
| T5 支付网关拒绝 | 余额/通道异常 | 快速失败，通知用户换退款方式 | 无（退款单未生成） |
| T6 CRM回写失败 | CRM 5xx | 写入补偿队列(MQ)，不影响用户侧 | 异步重试→失败告警→人工对账 |
| T7 通知失败 | 推送通道异常 | 写入通知重试队列 | 多通道降级: push→短信→站内信 |

### 2.5 Saga 补偿链设计

```
正向流程:  T1 → T2 → T3 → T4(确认) → T5(退款) → T6(CRM) → T7(通知)
补偿链:    C5(撤销退款) ← C6(无需) ← C7(无需)

关键原则:
1. T5之前的节点均为"读操作+校验"，无副作用，无需补偿
2. T5是唯一的资金写操作，一旦成功即不可逆（用户已收到退款承诺）
3. T6失败不触发T5补偿——CRM是辅助系统，走最终一致
4. 真正需要补偿的唯一场景：T5成功后发现是欺诈 → 走线下追回流程（超出系统自动化范围）
```

### 2.6 状态机设计

```
                    ┌──────────────────────────────────────────┐
                    │                                          │
  ┌─────────┐   ┌──▼──────┐   ┌─────────┐   ┌────────────┐  │
  │ CREATED │──▶│VERIFYING │──▶│CONFIRMED│──▶│ PROCESSING │──┼──▶ COMPLETED
  └─────────┘   └──┬──────┘   └────┬────┘   └─────┬──────┘  │
                    │               │               │         │
                    ▼               ▼               ▼         │
               SLOT_MISSING    REJECTED        FAILED ────────┘
               (追问中)        (不符合条件)     (可重试/人工)
                    │
                    ▼
               ABANDONED (超时未回复,30min TTL)
```

**中断恢复**：用户可在任何追问环节断开，30 分钟内回来继续（状态持久化在 DB）。超 30 分钟标记 ABANDONED，再来则新建。

### 2.7 场景验证结论

- ✅ 意图层三级路由（规则直接命中）
- ✅ 规划层 DAG 编排（含 HITL 节点）
- ✅ 执行层幂等 + Saga + 异步补偿
- ✅ 会话状态持久化与中断恢复

### 3.4 坐席匹配路由

| 匹配维度 | 权重 | 说明 |
|---------|------|------|
| 技能组匹配 | 必要 | 投诉组 / 售后组 / VIP组 |
| 当前负载 | 0.35 | 在线会话数 < 阈值 |
| 历史解决率 | 0.25 | 同类投诉的历史 CSAT |
| 语言/方言匹配 | 0.15 | 根据用户偏好/地域 |
| 最近服务过该用户 | 0.15 | 优先同一坐席（连续性） |
| 坐席情绪状态 | 0.10 | 避免分配给连续高压坐席 |

**排队策略**：
- 有空闲坐席 → 直接分配（目标 <15s）
- 无空闲 → 排队 + AI 持续安抚
- 排队超 60s → 升级到主管组 / 提供回拨
- 排队超 180s → VIP 自动升级最高优先级

| 异常场景 | 处理策略 | 兜底 |
|---------|---------|------|

| 坐席接入后用户离开 | 保留工单+上下文，24h 内回来路由同一坐席 | 超时主动回访 |
| 转人工后仍未解决 | 坐席再升级主管，触发 48h SLA 计时 | 超时告警+自动升级 |
| AI安抚阶段用户辱骂 | 不对抗，持续共情+快速转人工，全量日志保留 | 3 轮无实质诉求→温和提示 |

## 4. 场景3：大促流量峰值

### 4.1 场景描述

双11 开售瞬间，客服系统流量从日常 **500 QPS** 骤升至 **15,000 QPS**（30 倍），持续约 15 分钟后回落到 5,000 QPS（10 倍日常），维持 6 小时。主要涌入问题：价保咨询(40%)、物流催单(25%)、优惠规则咨询(20%)、退款/改单(15%)。

### 4.2 流量模型与分级

                     5000 ┤            ╭╯                          ╰────────────
                          │           ╭╯
                      500 ┤───────────╯

### 4.4 L2 限流策略详细设计

**分级限流（令牌桶）**：
- VIP/SVIP 用户：不限流（保障体验）
- 资金类会话（退款/支付）：不限流（业务关键）
- 普通咨询类：限流 50%（排队）
- 重复请求（同用户 1min 内）：直接拒绝

**排队机制**：
- 被限流请求进入 FIFO 队列
- 前端显示"当前排队XX人，预计等待XX秒"
- 队列深度上限：10000，超出直接拒绝+引导自助
- 队列 TTL：120s，超时释放+推送通知

**按租户配额**：
- 每个租户有预设 QPS 上限（SLA 合同约定）
- 超出配额部分排队或拒绝

## 5. 场景4：FAQ快速应答

### 5.1 场景描述

用户问"7天无理由退货需要什么条件？"——高频 FAQ 问题，日均命中数千次。通过三条路径的对比推演，验证模型路由与缓存设计的经济性和有效性。

### 5.2 三条路径总览

| 路径 | 触发条件 | 延迟 | 成本 | 覆盖率 | 精度 |
|------|---------|------|------|--------|------|
| **A: 极速** | 规则命中 + 缓存命中 | <100ms | ≈¥0 | 30% | 99%+ |
| **B: 标准RAG** | 分类器 + 检索 + 小模型 | 800-1500ms | ≈¥0.002 | 55% | 95%+ |
| **C: 深度推理** | LLM意图分析 + 多路RAG + 大模型 | 3000-5000ms | ≈¥0.05 | 15% | 90%+ |

### 5.3 路径A：极速命中（规则+语义缓存）

```
用户Query → 规则正则命中(conf>0.9) → 语义缓存查询 → 命中! → 直接返回

全链路: 50-100ms, 0次LLM调用, 成本≈¥0
```

**适用**：高频标准问题，问法清晰，历史已有高质量缓存答案。

### 5.4 路径B：标准RAG（分类器+检索+小模型）

```
用户Query → 规则部分命中(conf<0.9) → 分类器确认(conf>0.8)
         → 语义缓存miss → 混合检索(向量+BM25) → RRF融合 → Rerank精排
         → 小模型(7B)基于知识片段生成答案 → 返回(附来源引用)
         → 异步写入语义缓存

全链路: 800-1500ms, 1次小模型调用, 成本≈¥0.002
```

**适用**：明确单意图，有知识覆盖，但问法新颖或上下文特殊。

### 5.5 路径C：LLM深度推理（复合问题）

```
用户Query(复合/模糊) → 规则未命中 → 分类器多意图/低置信
                    → 升级到LLM做意图分析(函数调用)
                    → 拆解为多个子问题 → 多路RAG检索(各子问题独立检索)
                    → 大模型(70B+)整合推理 → 置信度检查
                    → 返回(附来源+免责+转人工入口)

全链路: 3000-5000ms, 2次大模型调用, 成本≈¥0.05
```

**适用**：复合意图、跨域问题、需推理综合的长尾场景。

### 5.6 模型路由决策引擎

```
路由决策树:

IF system_level >= L3:
    → 模板回复(零LLM)

ELIF conf_score > 0.9 AND intent_cnt == 1 AND cache_hit:
    → 缓存直返(零LLM)                               [路径A]

ELIF conf_score > 0.8 AND intent_cnt == 1 AND rag_score > 0.7:
    → 小模型(7B本地)生成                             [路径B]

ELIF conf_score < 0.7 OR intent_cnt > 1 OR complexity == HIGH:
    → 大模型(70B+/云端)推理                          [路径C]

ELSE (中间地带):
    → 先尝试小模型 → 自检置信度 → 不过则升级大模型
```

**成本控制约束**：
- 单会话 token 预算：普通用户 ≤4000，VIP ≤8000，SVIP 无限
- 大模型日调用配额：按租户 SLA 分配
- 超预算：降级小模型 → 降级模板 → 建议转人工
- 夜间(0-6点)：优先大模型（闲时成本低，提升质量）

### 5.7 语义缓存架构

```
双层缓存:

┌─────────────────────────────────────────────────┐
│  L1: 精确缓存 (Redis Hash)                       │
│                                                 │
│  key = hash(intent + sorted_slots)              │
│  命中率: ~20% (完全相同的意图+槽位)              │
│  TTL: 知识版本绑定(知识更新即失效)               │
└─────────────────────────────────────────────────┘
                    │ miss
                    ▼
┌─────────────────────────────────────────────────┐
│  L2: 语义缓存 (向量相似)                         │
│                                                 │
│  key = query_embedding (768d), index = HNSW     │
│  命中率: ~40% (加上L1共~60%)                    │
│  阈值: cosine_sim > 0.95 直返                   │
│        0.90-0.95 返回但标记"可能不精确"          │
│        < 0.90 不命中                            │
│  TTL: 7天 + LRU淘汰                             │
└─────────────────────────────────────────────────┘
```

**缓存治理规则**：
- 写入：仅写入 conf>0.85 且用户未踩的答案
- 失效：知识更新→对应 intent 缓存清除；用户踩→立即移除
- 防幻觉：缓存答案必须携带 source_id，知识 ID 失效则答案失效

### 5.8 RAG 检索质量保障

```
检索链路:
用户Query → Query改写(可选) → 混合检索 → RRF融合 → Rerank → 质量门控 → 输出

混合检索三路:
├── 向量检索 (语义相似，概念类问题)
├── BM25 (词法匹配，专有名词/政策编号)
└── 结构化查询 (精确字段，订单号/商品ID)

质量门控:
· top1_score > 0.6 → 正常回答
· 0.4 ≤ top1 < 0.6 → 回答标注"仅供参考"
· top1 < 0.4 → "暂无相关信息" + 转人工
· 知识过期 → 标注"政策可能已更新"
```

### 5.9 防幻觉全链路设计

| 层级 | 防幻觉措施 | 失败兜底 |
|------|-----------|---------|
| 检索层 | 质量门控(score阈值) + 知识时效校验 | 无据→不答 |
| 生成层 | Prompt约束"仅基于知识回答" + 禁止编造数字/时限 | 自检不通过→建议转人工 |
| 输出层 | 答案所有数字/条款核对知识原文 + 引用ID验证 | 无精确引用→降级表述 |
| 反馈层 | 用户踩→标记 + 质检幻觉检测 + 趋势监控 | 坏案例→知识修正→缓存清除 |

**特殊规则**：
- 涉及金额：必须有精确条款引用，否则不给数字
- 涉及时限：必须有官方文档依据，不确定给范围+免责
- 涉及法律：加免责声明，建议咨询专业人士

### 5.10 性能与成本模型

| 指标 | 路径A(30%) | 路径B(55%) | 路径C(15%) | 加权平均(日均10万会话) |
|------|-----------|-----------|-----------|---------------------|
| 端到端延迟 | 50-100ms | 800-1500ms | 3000-5000ms | P50:900ms P95:3200ms |
| LLM调用 | 0 | 1次(小) | 2次(大) | 日均8.5万次 |
| 单次成本 | ≈¥0 | ≈¥0.002 | ≈¥0.05 | 日均¥860 ≈ ¥2.6万/月 |
| 精度 | 99%+ | 95%+ | 90%+ | 加权95.5% |

**成本优化杠杆**：
- 语义缓存 60%→80%：月成本 ¥2.6万 → ¥1.6万（省38%）
- 路径A覆盖率 30%→50%：月成本再降 ¥0.4万
- Prompt压缩（摘要上文）：单次成本降30%

### 5.11 场景验证结论

- ✅ 推理层三级模型路由（零调用/小模型/大模型）
- ✅ 语义缓存双层设计与治理
- ✅ RAG 混合检索 + Rerank + 质量门控
